# Phase 1 friction log

Phase 1 drove one package (`packages/tictactoe`, spec #1, task #2) through the full loop
manually: issue → spec → task → implementation → tests → PR → CI → merge. This log records
the friction found in each step, as input for Phase 2 automation choices.

## Spec issue

- `gh issue create --body` bypasses the issue-form validation entirely: the YAML templates
  (`spec.yml`, `task.yml`) only enforce required fields in the web UI. A CLI/agent-driven
  flow reproduces the template structure by convention, not by contract. Phase 2 agents
  will need the structure enforced elsewhere (e.g. a bot that validates issue bodies).
- The form layout doesn't translate 1:1 to a markdown body; headings must be re-created by
  hand (`### Problem`, `### Acceptance criteria`, ...).

## Task issue

- Duplication: acceptance criteria are copied from the spec into the task, and later into
  the PR. Three hand-synced copies of the same contract. Automation should carry these
  forward mechanically.
- Nothing links a task to its branch/PR until the PR exists; the association is convention
  (`Part of #2` in commits, `Closes #2` in the PR).

## Scaffold (nx generator)

- Smooth overall: `nx g @nx/react:app` detected the TS-solution workspace and Bun, updated
  `nx.json`, root `tsconfig.json`, and `package.json` correctly, and installed with Bun
  (no stray npm lockfile).
- Generator gap: `tsconfig.app.json` gets `"lib": ["dom"]` but `tsconfig.spec.json` does
  not, so tests touching DOM types (`element.textContent`) fail `typecheck` against this
  repo's `lib: ["es2022"]` base. Fixed by adding `"lib": ["dom", "es2022"]` to
  `tsconfig.spec.json`.
- The generated `app.spec.tsx` uses `@testing-library/react` but the generator does not
  install `@testing-library/jest-dom`, so common matchers like `toHaveTextContent` are
  unavailable ("Invalid Chai property"). Worked around with plain `textContent` assertions.
- The scaffold ships marketing boilerplate (`nx-welcome.tsx`, ~800 lines) that must be
  deleted; noise in the diff and in review.

## Implementation + tests

- No friction. Pure-logic module (`game.ts`) + thin React UI kept tests fast and direct;
  vitest config generated by Nx worked unchanged (`watch: false` already set, so no CI
  hang risk).

## Local verification

- The CI gate is reproducible locally as one line
  (`bun run format:check && bun run typecheck && bun run test && bun run build`) — cheap
  insurance before pushing. Worth encoding as a single script or pre-push hook later.
- Nx flagged `typecheck` as a "flaky task" once (cache noise), an ad for Nx Cloud rather
  than a real failure.

## PR → CI → merge

- Same template gap as issues: `gh pr create --body` does not apply
  `pull_request_template.md`; the structure was reproduced by hand. Any non-web entry
  point (CLI, agent) relies on convention for the template contract.
- The local agent harness blocks `git push` (a deliberate guardrail), so the human
  performed the push and the agent resumed from there. Phase 2 must decide where pushes
  originate — a bot identity in CI, or a human gate like this one.
- CI passed in 18s; squash-merge went through cleanly under branch protection
  (0 approvals required while single-maintainer).
- `Closes #2` auto-closed the task issue, but nothing connects task completion back to
  the parent spec — the spec (#1) had to be closed manually. Spec lifecycle tracking is
  a candidate for automation.
- Acceptance criteria were now hand-copied for the third time (spec → task → PR).
  Confirmed as the most mechanical, automatable duplication in the loop.
